# 模块系统与路径别名

前端工程里最容易出现“编辑器能跳，运行时崩掉”的地方，就是模块系统和模块解析。

很多人对这部分的理解停留在：

- `import/export` 是 ESM
- `require/module.exports` 是 CommonJS
- `@/utils` 是路径别名

这些说法都没错，但太薄了。真正落到工程里，模块系统至少会影响这些事：

- 代码到底按什么语义运行
- 文件后缀是否必须显式写出来
- 打包器和 Node 如何找文件
- 路径别名只影响编辑器，还是运行时也能认
- `monorepo` 里边界到底按目录，还是按包

这篇就把这条线完整讲清楚。

## 一、为什么模块系统是工程化问题，不只是语法问题

模块系统的表面是导入语法，底层其实是依赖边界。

如果模块语义没统一，项目会出现很典型的症状：

- 编辑器里没问题，Node 执行时报 `Cannot find module`
- 本地能跑，构建产物到了服务器上报 ESM/CJS 错误
- 路径别名在 TS 里能识别，运行时却不认
- 在 `monorepo` 里跨目录直连源码，依赖图失效

所以模块系统不是“风格选择”，而是工程边界的一部分。

## 二、先把 ESM 和 CommonJS 讲清楚

### 1. ESM 是现代模块语义

典型写法：

```ts
import { sum } from "./utils"
export function calc() {}
```

特点：

- 语义静态，利于分析和 Tree Shaking
- 更贴近浏览器原生模块标准
- 现代前端工具链基本都围绕它构建

### 2. CommonJS 是 Node 历史模块系统

典型写法：

```js
const { sum } = require("./utils")
module.exports = { calc }
```

特点：

- 历史兼容性强
- 在老项目和部分 Node 生态里仍然大量存在

### 3. 问题不在于谁先进，而在于语义是否统一

如果一个项目里：

- 源码按 ESM 写
- 运行时按 CommonJS 理解
- 构建产物又混出另一套行为

那你遇到的问题往往不是某一行 import 写错，而是整个模块语义没统一。

## 三、`module` 和 `moduleResolution` 分别管什么

这是 TypeScript 模块系统里最容易混淆的一组配置。

### 1. `module`

它决定 TypeScript 如何理解和输出模块语义。

常见值：

- `ESNext`
- `CommonJS`
- `NodeNext`

### 2. `moduleResolution`

它决定导入路径怎么被解析成实际文件。

常见值：

- `Bundler`
- `NodeNext`
- `Node16`
- `Node`

### 3. 为什么这两者必须一起看

因为一个管“你想用什么语义”，一个管“文件怎么找”。

如果只改其中一个，经常会出现：

- TS 能编，但 Node 不能跑
- 本地开发能跑，构建不一致
- 后缀规则前后不统一

## 四、现代前端项目为什么通常用 `Bundler`

如果你的项目使用：

- `vite`
- `tsx`
- `tsup`
- `esbuild`

那通常更推荐：

```json
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "Bundler"
  }
}
```

### 1. `Bundler` 的直观好处

- 更接近现代打包器行为
- 导入时通常不用执着于 Node ESM 那套显式后缀要求
- 对前端应用和现代 Node 工具链更省心

### 2. 它适合哪些场景

- React / Vite 应用
- 使用 `tsx` 跑的 Node 项目
- `tsup` 打包的库
- 绝大多数现代前端工程

## 五、什么时候会用 `NodeNext` / `Node16`

如果你做的是更严格的 Node ESM 项目，并且运行时真的遵守 Node 原生模块解析规则，那么更常见的是：

```json
{
  "compilerOptions": {
    "module": "NodeNext",
    "moduleResolution": "NodeNext"
  }
}
```

这时要注意，Node 对 import 路径和后缀要求通常更严格。

### 1. 为什么很多人会觉得麻烦

因为在这种模式下，你更容易碰到：

- 导入时后缀问题
- `.js`、`.mjs`、`.cjs` 语义差异
- 包导出字段和运行行为对不齐

### 2. 为什么前端项目通常不需要这么“原生”

因为前端应用本来就依赖构建工具，它们不是直接把源码裸交给 Node 去跑。所以绝大多数前端项目直接用 `Bundler` 更务实。

## 六、`package.json` 里的 `type` 字段到底有什么用

这是 Node 运行时语义的重要开关。

```json
{
  "type": "module"
}
```

加上之后，Node 会把 `.js` 默认按 ESM 理解。

如果不加，默认更偏 CommonJS。

### 1. 为什么这个字段经常被忽略

因为很多项目开发态一直依赖 `vite`、`tsx` 这类工具，直到某天用 `node dist/index.js` 或线上环境执行时，才发现语义对不上。

### 2. 一条务实建议

如果你的项目整体就是现代 ESM 风格，就尽量统一：

- `package.json` 明确 `type: module`
- TS 配置和构建产物围绕 ESM 组织

不要源码用 ESM，运行时再偷偷退回 CommonJS 语义。

## 七、路径别名 `paths` 到底解决什么问题

很多人最先接触模块配置，就是因为烦透了：

```ts
import { formatDate } from "../../../../utils/date"
```

所以会配：

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

然后写成：

```ts
import { formatDate } from "@/utils/date"
```

这当然更好读，但你要知道它真正解决的是两类问题。

### 1. 导入可读性

让导入语义更稳定，不再被目录层级拖着走。

### 2. 目录重构成本

当你重构某个页面或组件目录时，不会因为相对路径层级变化导致一堆 import 同时改写。

## 八、路径别名不是架构边界本身

这是一个很关键的认知。

即使你用了：

```ts
import { request } from "@/utils/request"
```

也不代表你的项目边界就清晰了。

如果：

- 页面层能随意 import 数据层私有实现
- 组件层能直接引用某个页面的内部状态
- 所有目录都能互相直连

那路径别名只是把混乱写得更像“工程化”而已。

所以路径别名应该被看作：

- 导入层面的优化
- 为架构治理服务的辅助工具

而不是架构治理本身。

## 九、`paths` 只配 TS 还不够，运行时和构建时也要认

这是非常常见的坑。

你在 `tsconfig.json` 里写了：

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

编辑器可能立刻就能跳转，但这不代表运行时一定认识这个别名。

### 1. 为什么会这样

因为：

- TypeScript 认识别名
- 编辑器认识别名
- 运行器和构建器不一定认识别名

### 2. 在现代工具里通常怎样

- `tsx` 4.x 对 `paths` 支持较好
- `vite` 通常需要在配置里同步 alias 或通过插件处理
- 测试工具也可能要单独同步解析规则

所以一个别名配置是否真正成立，要看整条链路：

- TypeScript
- 编辑器
- 开发运行器
- 构建器
- 测试环境

## 十、`allowImportingTsExtensions` 是干什么的

有些项目喜欢这样写：

```ts
import { helper } from "./helper.ts"
```

默认 TypeScript 对此比较保守。如果你明确需要这种写法，可以开启：

```json
{
  "compilerOptions": {
    "allowImportingTsExtensions": true,
    "noEmit": true
  }
}
```

这个选项更适合：

- 不依赖 `tsc` 输出 JS
- 运行和构建由现代工具处理

如果你依赖 `tsc` 直接产出 JS，就要非常清楚构建后的路径和后缀语义是否仍然成立。

## 十一、模块边界在 Monorepo 里会变得更重要

单项目里，模块更多是目录级组织问题。到了 `monorepo`，模块边界会升级成包边界问题。

### 1. 单项目里的正确做法

在一个应用内部，可以使用：

```ts
import { Button } from "@/components/Button"
```

这是应用内部的路径组织。

### 2. Monorepo 里的错误做法

不要这样跨目录直连另一个包的源码：

```ts
import { sum } from "../../../packages/utils/src/index"
```

或者更隐蔽地通过某种别名绕过去。

### 3. Monorepo 里的正确做法

应该声明工作区依赖，然后按包名导入：

```ts
import { sum } from "@repo/utils"
```

这样做的好处非常实际：

- 依赖关系可被工作区工具正确识别
- 构建顺序能按依赖图推导
- 包边界清晰
- 后续发包、缓存、测试隔离才有基础

## 十二、为什么“跨目录 import 源码”是非常危险的反模式

很多人觉得：

“反正都在一个仓库，直接 import 源码也没什么。”

这是典型短期省事、长期埋雷的做法。

它会带来这些后果：

- 工作区依赖图失真
- `turbo` / `nx` 这类任务编排工具无法准确推导关系
- 包的构建和测试边界被破坏
- 以后拆分、发版、缓存几乎都会出问题

一句话：

在 `monorepo` 里，目录不是依赖边界，包才是依赖边界。

## 十三、前端项目里比较务实的模块策略

如果你现在是一个现代前端项目，可以先按这套理解。

### 1. 应用项目

- `module: ESNext`
- `moduleResolution: Bundler`
- 用路径别名改善应用内部导入体验
- 不要把别名当成跨包能力

### 2. Node 项目

如果开发运行由 `tsx` 或打包器兜底，通常也可以用 `Bundler`。

如果你是严格 Node ESM 原生运行，再考虑 `NodeNext`。

### 3. Monorepo 项目

- 应用内部继续使用自己的别名，例如 `@/*`
- 跨包依赖统一走包名，例如 `@repo/ui`
- 不要拿应用内 alias 去跨目录引用别的包源码

## 十四、最常见的误区

### 1. “能 import 就说明配置没问题”

错。编辑器能跳转，不代表运行时和构建时也能识别。

### 2. “路径别名就是工程化”

错。路径别名只是导入体验优化，不是架构治理本身。

### 3. “Monorepo 里反正都在一个仓库，直接跨目录引用就行”

错。这会直接破坏包边界。

### 4. “ESM / CJS 无所谓，能跑就行”

短期可能是这样，长期通常会在构建、部署、测试或第三方库兼容阶段爆出来。

## 十五、一个更靠谱的判断顺序

当你配置模块系统时，建议按这个顺序想：

1. 项目的真实运行环境是什么
2. 我想统一成 ESM 还是 CommonJS
3. 开发和构建工具是否主要围绕打包器语义
4. 是否需要路径别名
5. 是否处在 `monorepo` 中，需要区分应用内部别名和跨包导入

顺序不要反。很多项目一上来就先配 `@/*`，但最根本的运行语义还没定下来。
